# Execution Engine Architecture

**ENRS Unified Communications Platform**  
Version 1.0 · 2026-07-21

---

## Purpose

The Execution Engine is the platform's voice layer — the combination of FreeSWITCH, the ESL connection (`eslService.js`), the Conference Manager, and the Lua script runtime. It converts high-level call instructions into actual audio sessions and emits events back to the application.

This document describes the current implementation (what exists now) and the target architecture (after Wave 4). The two are intentionally similar — the main change is wrapping ESL access behind the Provider interface rather than calling it from business modules directly.

---

## Components

### `eslService.js` — ESL Connection Manager

Owns the single persistent ESL TCP connection to FreeSWITCH. Responsibilities:

- Connect to FS ESL on startup (`FS_HOST:FS_ESL_PORT` env vars)
- Subscribe to `ALL` events
- Route events to registered handlers
- Provide `eslCommand(cmd)` — executes a FreeSWITCH API command synchronously
- Maintain `eslEventBus` — Node.js EventEmitter for `CHANNEL_ANSWER`, `CHANNEL_HANGUP`, `CONFERENCE_MEMBER_ADD`, `CONFERENCE_MEMBER_DEL`, `RECORD_STOP`

All FreeSWITCH ESL access MUST go through `eslService.js`. No other file may open an ESL connection or maintain its own event subscriptions.

**Current violation to resolve (Wave 1):** `ersRingService.js` calls `eslCommand('bgapi originate …')` directly with inline variable block construction. This moves to `outboundRouter.js` → `freeswitchProvider.js` in Wave 1.

---

### `conferenceManager.js` — Conference Room Manager

Owns conference room naming, profile selection, and conference string assembly.

**Current responsibilities:**
- `resolveConferenceRoom(config, slot)` → room name (e.g., `ers_42_p`, `ers_42_s`)
- `getConferenceProfile(config)` → validated FreeSWITCH conference profile name
- `getConferenceString(config, slot)` → `"room@profile"` for use in `&conference()` action
- `handleConferenceCreated(confName)` → triggers ERS recording (BOUNDARY VIOLATION — see F3 below)
- `handleFirstParticipant(confName, uuid)` → sets FS moderator flag (platform-level, correct)
- `maybeAutoRecord()`, `startAutoRecording()` — ERS business logic (BOUNDARY VIOLATION)

**Target responsibilities (Wave 3):**
- `resolveConferenceRoom`, `getConferenceProfile`, `getConferenceString` — unchanged
- `handleFirstParticipant` — unchanged (platform hook, correct)
- `handleConferenceCreated` — emit an event. ERS module listens and decides whether to record.
- `maybeAutoRecord`, `startAutoRecording` — move to ERS module or ERS-specific RecordingService

**F3 Boundary Violation:** `conferenceManager.js` currently queries `ers_incidents` and `ers_configurations` directly. A platform service must not access business module tables. This is fixed in Wave 3 by replacing the direct call with an EventEmitter pattern — the platform fires `conference:created` and the ERS module handles it.

### `freeSwitchPathService.js` — Filesystem Path Registry

Single source of truth for all paths on the FreeSWITCH host:

```javascript
{
  dialplanDir:        process.env.FS_DIALPLAN_DIR    || '/etc/freeswitch/dialplan/default/',
  luaDir:             process.env.FS_LUA_DIR         || '/usr/share/freeswitch/scripts/',
  gatewayDir:         process.env.FS_GATEWAY_DIR     || '/etc/freeswitch/sip_profiles/external/',
  recordingsDir:      process.env.FS_RECORDINGS_DIR  || '/var/lib/freeswitch/recordings/',
  autoloadConfigsDir: process.env.FS_AUTOLOAD_CONFIGS|| '/etc/freeswitch/autoload_configs/',
}
```

Every file that needs a FreeSWITCH path reads from this service. Hard-coded paths anywhere else are a bug.

## Lua Script Runtime

Lua scripts run inside FreeSWITCH. They are generated by the Node.js application and deployed to `FS_LUA_DIR`. They have no direct database access — all business data comes from the Internal API (`/api/v1/internal/*`).

### Current Lua scripts

| Script | Trigger | Purpose |
|---|---|---|
| `ers_conference_bridge.lua` | Inbound ERS call | Lookup config, bridge responders, manage queue |
| `ens_blast_trigger.lua` | Inbound ENS blast number | Start a blast — PIN verify, record, start campaign |
| `ens_playback_handler.lua` | Inbound ENS playback number | Play latest recording or no-pending message |
| `ivr_executor.lua` | Inbound IVR number | Execute IVR flow graph node by node |
| `dtmf_handler.lua` | DTMF events during call | Handle digit collection for menus |

### IVR Lua — generated per flow

The IVR deployment engine generates a unique Lua script per published IVR flow version. These scripts use the generic `ivr_executor.lua` as the runtime and embed the flow as a JSON constant. This is the correct architecture — the executor does not need to fetch the flow from the API at runtime.

```lua
-- Generated: ivr_flow_abc123.lua
local FLOW = [[ { ... flow JSON ... } ]]
require("ivr_executor").run(FLOW)
```

### Lua → Internal API Contract

Lua speaks HTTP only. `io.popen(curl ...)` is the HTTP client. Every Lua script follows this pattern:

1. One `GET /internal/<module>/lookup` call at session start → gets full config in one response
2. Zero or more `POST /internal/<module>/event` calls → records state transitions
3. No session state stored in Lua — stateless per call

The lookup endpoint must return everything a Lua script needs for the entire call. No mid-call config queries.

## FreeSWITCH Event Flow

```
FS ESL event stream
     │
     ▼
eslService.js (eslEventBus)
     │
     ├── CHANNEL_ANSWER  → campaignEngine.onCallAnswer(uuid)
     │                  → ersRingService.onCallAnswer(uuid)
     │
     ├── CHANNEL_HANGUP  → campaignEngine.onCallHangup(uuid, cause)
     │                  → ersRingService.onCallHangup(uuid, cause)
     │
     ├── CONFERENCE_MEMBER_ADD → conferenceManager.handleFirstParticipant()
     │                        → [Wave 3: emit 'conference:member:joined']
     │
     ├── CONFERENCE_MEMBER_DEL → [Wave 3: emit 'conference:member:left']
     │
     └── RECORD_STOP           → recordingController.handleRecordStop()
```

After Wave 4, all event routing goes through the Provider Layer:

```
FS ESL event stream
     │
     ▼
eslService.js
     │
     ▼
freeswitchProvider.js (translates to standard events)
     │
     ├── 'answer'  → communicationEngine.onSessionAnswer(sessionUuid)
     ├── 'hangup'  → communicationEngine.onSessionHangup(sessionUuid, standardCode)
     └── 'media'   → conferenceManager.onConferenceEvent(type, data)
```

## `enrs_session_uuid` Channel Variable

Every outbound call MUST set the channel variable `enrs_session_uuid` equal to the Communication Session's `session_uuid`. This is the correlation key between a FreeSWITCH channel and a platform session.

When a `CHANNEL_HANGUP` event arrives, the application looks up the session by `enrs_session_uuid` from the event headers — not by FreeSWITCH UUID. This decouples the platform from FreeSWITCH's internal UUID scheme.

**This variable must be set in `outboundRouter.js` only.** Any code that calls `eslCommand('bgapi originate …')` inline without setting `enrs_session_uuid` is a bug.

## ESL Command Types

| Type | When to use | Behavior |
|---|---|---|
| `bgapi originate` | ENS blast, ERS ring-all (many calls) | Non-blocking; FS queues the call |
| `api originate` | Single call requiring immediate UUID | Blocking; waits for call setup |
| `api uuid_kill <uuid>` | Cancel an active call | Immediate |
| `api reloadxml` | After gateway XML change | Reloads FS config |
| `api sofia profile external rescan` | After gateway deployment | Re-registers gateways |
| `api conference <room> list` | Conference status check | Blocking list |

The choice between `bgapi` and `api originate` belongs in `outboundRouter.js` via the `async` parameter on the `CommunicationRequest`.

## ESL Connection Resilience

`eslService.js` reconnects automatically on disconnect. All pending command promises reject on disconnect and the caller must retry.

Known gap (Wave 2): No observable health check is exposed. A `/api/v1/health` endpoint should report ESL connection state so deployment tooling can verify the backend is live.
