# XML Generation Pipeline

## Overview

Deploying an IVR flow writes two files to the FreeSWITCH filesystem:

1. **`ivr_executor.lua`** — the single generic Lua script that handles all flows.
2. **`enrs_ivr.xml`** — the FreeSWITCH dialplan XML mapping dialled numbers to the Lua script.

Both files are written on every deploy. The Lua file is overwritten (idempotent — same content for all flows on a tenant). The XML file is regenerated from all bound numbers across all flows, not just the one being deployed.

---

## Path Service — `freeSwitchPathService`

All filesystem paths are resolved through the `FreeSwitchPathService` class (`backend/src/services/freeSwitchPathService.js`). This is a singleton exported as `fsPathService`. No deployment or audio code accesses `process.env.FS_*` directly.

### Environment Variables

| Variable | Default | Purpose |
|---|---|---|
| `FS_BASE_DIR` | `/usr/share/freeswitch` | Base for script and sound paths |
| `FS_CONF_DIR` | `/etc/freeswitch` | FreeSWITCH configuration root |
| `FS_DIALPLAN_DIR` | `/etc/freeswitch/dialplan` | Dialplan XML directory |
| `FS_SCRIPT_DIR` | `${FS_BASE_DIR}/scripts` | Lua scripts directory |
| `FS_SOUND_DIR` | `${FS_BASE_DIR}/sounds` | FreeSWITCH sounds directory |
| `FS_RECORDINGS_DIR` | `/var/lib/freeswitch/recordings` | Recording base directory |
| `FS_RECORDING_DIR` | (same) | Alternative spelling accepted |
| `ENRS_REC_DIR` | (same) | Third alternative spelling |
| `FS_SIP_PROFILE_DIR` | `/etc/freeswitch/sip_profiles` | SIP profile directory for gateways |
| `FS_STORAGE_DIR` | `/var/lib/freeswitch/storage` | FreeSWITCH storage directory |
| `FS_DB_DIR` | `/var/lib/freeswitch/db` | FreeSWITCH database directory |
| `FS_LOG_DIR` | `/var/log/freeswitch` | FreeSWITCH log directory |

`FS_RECORDINGS_DIR` takes priority over `FS_RECORDING_DIR`, which takes priority over `ENRS_REC_DIR`.

### Path Service Methods

```
getConfigDir()       → FS_CONF_DIR
getDialplanDir()     → FS_DIALPLAN_DIR
getScriptDir()       → FS_SCRIPT_DIR
getSoundDir()        → FS_SOUND_DIR
getEnrsSoundDir()    → FS_SOUND_DIR/enrs
getRecordingDir()    → FS_RECORDINGS_DIR
getErsRecordingDir() → FS_RECORDINGS_DIR/ers
getEnsRecordingDir() → FS_RECORDINGS_DIR/ens
getIvrRecordingDir() → FS_RECORDINGS_DIR/ivr
getManualRecordingDir() → FS_RECORDINGS_DIR/manual
getSipProfileDir()   → FS_SIP_PROFILE_DIR
getExecutorLuaFile() → FS_SCRIPT_DIR/ivr_executor.lua
getIvrDialplanFile() → async — depends on detectDialplanTargetDir()
```

## `detectDialplanTargetDir` — Layout Auto-Detection

FreeSWITCH installations use two different dialplan layout conventions. Writing to the wrong directory results in a silently-dead extension: `reloadxml` reports `+OK` but the extension is never reachable because the file declares a sibling `<context name="default">` that is never merged with the one FreeSWITCH routes calls through.

`detectDialplanTargetDir()` inspects `default.xml` to determine which layout is in use. It is called on every deploy (not cached) because the FreeSWITCH configuration on a box can change between deployments.

### Return Value

```js
{ dir: string, nested: boolean }
```

- `dir`: the directory to write `enrs_ivr.xml` into.
- `nested`: whether the file must be written as a bare fragment (no `<context>` wrapper) or as a complete self-contained XML document.

### Detection Cases

**Case 1 — FS_DIALPLAN_DIR contains `default.xml` with a nested include:**

```xml
<context name="default">
  <X-PRE-PROCESS cmd="include" data="default/*.xml"/>
</context>
```

`dir = FS_DIALPLAN_DIR/default` (or whatever the `data` pattern resolves to), `nested = true`. The generated file must NOT declare its own `<context>` wrapper because it will be spliced inside an already-open one.

**Case 2 — FS_DIALPLAN_DIR IS the nested target directory:**

The env var was set directly to `...dialplan/default/` instead of `...dialplan/`. The parent directory's `default.xml` is checked: if its include resolves back to this directory, `nested = true`, `dir = FS_DIALPLAN_DIR`.

**Case 3 — Flat layout:**

No nested include found in either own `default.xml` or parent `default.xml`. `nested = false`, `dir = FS_DIALPLAN_DIR`. The generated file must include its own `<context name="default">` wrapper.

## IVR Dialplan XML Generation

`backend/src/utils/xmlGenerator.js` exports `generateDialplanXml(bindings, { nested, testMode })`.

### Input — `bindings`

Array of active number bindings from the database:

```js
[
  { number: '1800EMERGENCY', flow_uuid: 'abc-123', flow_name: 'Main IVR', version_number: 5 },
  { number: '3010', flow_uuid: 'def-456', flow_name: 'Security', version_number: 2 },
  // ...
]
```

### Dialplan XML Structure (nested layout)

When `nested = true`, the file is a bare fragment:

```xml
<!-- Generated by ENRS IVR Builder. DO NOT EDIT MANUALLY. -->
<!-- Bound numbers: 2 | Generated: 2026-07-20T10:30:00.000Z -->

<extension name="enrs_ivr_1800EMERGENCY">
  <condition field="destination_number" expression="^1800EMERGENCY$">
    <action application="lua" data="ivr_executor.lua"/>
  </condition>
</extension>

<extension name="enrs_ivr_3010">
  <condition field="destination_number" expression="^3010$">
    <action application="lua" data="ivr_executor.lua"/>
  </condition>
</extension>
```

When `nested = false`, the file wraps the extensions in a context:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<document type="freeswitch/xml">
  <section name="dialplan">
    <context name="default">
      <extension name="enrs_ivr_1800EMERGENCY">
        ...
      </extension>
    </context>
  </section>
</document>
```

### Extension Naming

Each extension is named `enrs_ivr_<number>`. This name is used by `verifyExtensionLoaded()` to confirm the extension appears in the live dialplan after reload.

### Test Mode

If `testMode.enabled` is true and a flow's UUID is in `testMode.testFlowUuids`, an additional `<condition>` is prepended that requires `Caller-Caller-ID-Number` to match `testMode.callerId`. This restricts the extension to test calls only without affecting production numbers.

## Lua Executor Deployment

`deployLuaExecutor()` in `deploymentEngine.js`:

1. Calls `generateIvrExecutorLua({ apiBase, apiKey, ttsEngine })` from `luaGenerator.js`.
2. Writes the result to `fsPathService.getExecutorLuaFile()`.
3. Sets file permissions to `0o644` (readable by the `freeswitch` process).

The generated file is the same for every flow and every tenant — it fetches the flow graph at call time using the destination number as the lookup key. There is never a per-flow or per-tenant Lua file.

## Gateway XML Generation

SIP gateway XML fragments are generated by `backend/src/utils/gatewayXmlGenerator.js` and deployed by `backend/src/services/gatewayDeployment.js`.

Each configured SIP gateway produces an XML fragment written to `FS_SIP_PROFILE_DIR/<profile_name>/<gateway_name>.xml`:

```xml
<gateway name="my-sip-provider">
  <param name="username" value="trunk01"/>
  <param name="realm" value="sip.provider.com"/>
  <param name="from-domain" value="sip.provider.com"/>
  <param name="password" value="***"/>
  <param name="proxy" value="sip.provider.com:5060"/>
  <param name="register" value="true"/>
</gateway>
```

After writing gateway files, `bgapi sofia profile <profile> rescan` is issued to reload the SIP profile without a full FreeSWITCH restart.

## `resolveMediaPath` — Audio URI Resolution

`fsPathService.resolveMediaPath(mediaUri)`:

```
/media/welcome.wav  →  ${FS_SOUND_DIR}/enrs/welcome.wav
/media/prompts/hold.wav  →  ${FS_SOUND_DIR}/enrs/prompts/hold.wav
```

Returns `null` if the URI does not start with `/media/`. Used by:
- The deployment pipeline's audio validation step (checks the resolved path exists on disk before deploying).
- The Lua executor's `resolve_audio()` function at call time.

The inverse operation `fsPathService.toMediaUri(fsPath)` converts an absolute sound path back to a `/media/` URI for display in the UI.

## Complete Deployment Pipeline

`deployFlow(flowUuid, { deployedBy, tenantId })` in `deploymentEngine.js` runs these steps in order. Each step is wrapped by `runStep(report, name, fn)` which records start time, end time, status, and any error in the deployment report.

| Step | Name | Action |
|---|---|---|
| 1 | `fetch_published_version` | Queries `ivr_flows JOIN ivr_flow_versions` for the latest version. Fails if no published version exists. |
| 2 | `validate_graph` | Calls `validateGraph(graph, tenantId)` from `ivrGraphValidator.js`. Checks: reachability of all nodes from `entry_node_id`, ERS/ENS config IDs exist in DB and match tenant, `next`/`true_node`/`false_node`/`target_node_id` references point to existing nodes. Fails hard if any errors; logs warnings. |
| 3 | `validate_audio_files` | For every node with `audio_url`, `play_audio_url`, or `prompt_audio_url` starting with `/media/`, checks that `resolveMediaPath(uri)` exists on disk. Issues warnings only — does not block deployment. |
| 4 | `ensure_directories` | Creates all required directories with `fs.mkdir(..., { recursive: true })`: script dir, dialplan dir, dialplan target dir, ENRS sound dir, IVR recording dir. |
| 5 | `deploy_lua_executor` | Writes `ivr_executor.lua` to `getExecutorLuaFile()`. Sets permissions to `0o644`. |
| 6 | `deploy_dialplan_xml` | Queries all active bound numbers across all flows. Detects dialplan layout via `detectDialplanTargetDir()`. Generates and writes `enrs_ivr.xml`. Sets permissions to `0o644`. |
| 7 | `reloadxml` | Sends `bgapi reloadxml` via ESL. If ESL is offline, records a warning and sets `eslConnected = false`. Does not fail the deployment. |
| 8 | `verify_extension_loaded` | Skipped if `eslConnected = false`. Queries bound numbers for this flow. For each number, calls `verifyExtensionLoaded("enrs_ivr_<number>")` which runs `xml_locate dialplan context name default` and scans the output. Retries 3 times with 500 ms delay. Fails the deployment if any extension is not found. |
| 9 | `record_deployment` | Updates `ivr_flows.last_deployed_at`, `last_deployment_status`, `last_deployed_version`. Inserts a row in `ivr_flow_deployments` with the full report JSON. |

On any step failure, the deployment is marked `'failed'` in the report, `ivr_flows.last_deployment_status = 'failed'` is recorded, and the report is inserted in `ivr_flow_deployments` with `status = 'failed'` and the error message.

## `verifyExtensionLoaded` — Dialplan Verification

```js
export async function verifyExtensionLoaded(
  extensionName,
  { attempts = 3, delayMs = 500, locateFn = xmlLocateDefaultContext } = {}
)
```

Internally calls:

```
bgapi xml_locate dialplan context name default
```

This is the correct 4-argument form of `xml_locate`. The 2-argument form (`xml_locate dialplan default`) is not a valid invocation and reliably produces false-negative "not found" results.

The `locateFn` parameter is injectable for unit testing — production callers never pass it.

Returns:

```js
{ loaded: boolean, raw: string, attempts: number, error?: string }
```

If `loaded = false` after all retries, the deployment controller surfaces the diagnostic message:

```
Deployed files were written but FreeSWITCH did not load the extension(s) (enrs_ivr_3010).
Check dialplan/default.xml's include pattern on this box —
run GET /deployment/diagnostics to see the resolved include chain.
```

## `redeployAll` — Rebind After Number Changes

When a number is bound or unbound from a flow, `redeployAll()` is called automatically. This function:

1. Queries all flows that have at least one active bound number.
2. Calls `deployDialplanXml()` — regenerates and writes `enrs_ivr.xml` (Lua is unchanged).
3. Sends `bgapi reloadxml`.

This ensures that a newly bound number is immediately live and an unbound number is immediately removed from the live dialplan. The Lua executor is not regenerated in this path because it does not contain per-number information.
