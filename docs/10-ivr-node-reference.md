## `play` — Play Audio

**Category:** Audio
**Port Strategy:** `next` (single successor)
**Terminal:** No

### Purpose

Plays a pre-recorded audio file from the media library to the caller. Use this node for greetings, informational messages, and announcements where a specific recorded file (not synthesized speech) is required.

### Configuration Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `audio_url` | string | One of `audio_url` or `audio_file_id` required | Local media URI in the format `/media/filename.wav`. Resolved at Lua runtime to `${FS_SOUND_DIR}/enrs/filename.wav`. No external URLs are permitted (SSRF prevention). |
| `audio_file_id` | integer | One of `audio_url` or `audio_file_id` required | ID from the `media_files` table. The absolute path is resolved at Lua generation time. |
| `next` | node_ref | Required | ID of the next node. |

### Variables Supported

`audio_url` supports `${varname}` interpolation at Lua runtime, allowing dynamic audio selection based on session state.

### Variables Produced

None.

### Validation Rules

- At least one of `audio_url` or `audio_file_id` must be set (enforced by `superRefine` in `AnyNodeSchema`).
- `audio_url` must start with `/media/` (enforced by the `localAudioUrl` Zod refinement — no absolute filesystem paths, no external URLs).
- `audio_file_id` must reference an existing, active `media_files` row belonging to the same tenant (DB FK check at publish time).

### Generated Lua

```lua
local function exec_play(s, node)
  local f = resolve_audio(node.audio_url)
  if f then s:streamFile(f) end
  return node.next
end
```

The `resolve_audio()` helper maps `/media/filename.wav` to the absolute filesystem path using the `FS_SOUND_DIR` environment variable.

### Runtime Behavior

1. `resolve_audio(node.audio_url)` maps the `/media/` URI to an absolute path.
2. If the path resolves to a non-nil value, `session:streamFile(path)` is called — this blocks until playback completes or the caller hangs up.
3. If `session:ready()` returns false during playback (caller disconnected), FreeSWITCH exits the Lua script cleanly.
4. On completion, the function returns `node.next` and the executor advances to the next node.

### Database Updates

None.

### ESL Events

None generated by this node. FreeSWITCH internally generates `CHANNEL_EXECUTE` / `CHANNEL_EXECUTE_COMPLETE` events for the `playback` application, visible in ESL trace logs.

### API Calls

None at runtime. `audio_file_id` path resolution occurs at Lua generation time (deploy step), not during the call.

### Common Mistakes

- Using an absolute filesystem path instead of the `/media/`-prefixed URI. Use `/media/welcome.wav`, not `/var/lib/freeswitch/sounds/enrs/welcome.wav`.
- Omitting both `audio_url` and `audio_file_id`. The validator rejects this at save time.
- Referencing a `media_file_id` belonging to a different tenant or organization. The DB FK check at publish time will return an error.
- Referencing a file that has not been deployed to FreeSWITCH. Upload and deploy audio files via Deployment → Audio Library before referencing them in flows.

### Debugging Steps

1. Check deployment logs for path resolution errors during the Lua generation step.
2. Verify the file exists on the FreeSWITCH host: `ls ${FS_SOUND_DIR}/enrs/filename.wav`
3. Check FreeSWITCH logs for `streamFile` errors: `fs_cli -x "log 7" | grep streamFile`
4. Confirm `media_files.is_deployed = true` and `media_files.deleted_at IS NULL` for the referenced file.
5. Check that `FS_SOUND_DIR` is set correctly in the backend `.env`.

### Expected Log Output

```
[ivr_executor] play node: streaming /usr/share/freeswitch/sounds/enrs/welcome.wav
```

### Failure Scenarios

| Failure | Symptom | Recovery |
|---|---|---|
| File not found on filesystem | FreeSWITCH logs "Failed to open file"; caller hears silence; flow continues to `next` | Re-deploy the audio file |
| Wrong audio format | FreeSWITCH decoder error in logs; caller hears silence; flow continues | Re-upload as 8kHz 16-bit mono PCM WAV |
| `audio_url` is empty string | `resolve_audio` returns nil; node is silently skipped | Set `audio_url` or `audio_file_id` |

## `say` — Say (TTS)

**Category:** Audio
**Port Strategy:** `next` (single successor)
**Terminal:** No

### Purpose

Synthesizes and speaks text to the caller using FreeSWITCH's TTS engine. Use this node when recorded audio is not available or when the message text must be dynamic (with variable interpolation).

### Configuration Fields

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `text` | string (1–1000 chars) | Yes | — | The text to speak. Supports `${varname}` interpolation. |
| `language` | string | No | `en-US` | BCP-47 language tag. Supported: `en-US`, `en-AU`, `en-GB`, `es-ES`, `fr-FR`, `de-DE`. |
| `voice` | string | No | — | Voice name for the configured TTS engine (e.g., `Joanna` for Amazon Polly). Ignored if the engine does not support named voices. |
| `next` | node_ref | Yes | — | ID of the next node. |

### Variables Supported

`text` supports `${varname}` interpolation at Lua runtime. Example: `"You have reached the ${ens_configuration_id} notification line."`

### Variables Produced

None.

### Validation Rules

- `text` must be non-empty and no longer than 1000 characters.
- `language` defaults to `en-US` if omitted.
- `next` is required.

### Generated Lua

```lua
local function exec_say(s, node)
  speak(s, interp(s, node.text))
  return node.next
end
```

The `speak()` helper calls `session:execute("speak", TTS_ENGINE .. "|" .. voice .. "|" .. text)` where `TTS_ENGINE` is read from the `ENRS_TTS_ENGINE` environment variable (default: `flite`).

The `interp()` helper resolves `${varname}` references against the current session's channel variables.

### Runtime Behavior

1. `interp(s, node.text)` resolves any `${varname}` placeholders in the text string.
2. `speak(s, resolved_text)` executes the FreeSWITCH `speak` application with the configured TTS engine.
3. The `speak` application blocks until synthesis and playback complete, or until the caller disconnects.
4. Returns `node.next`.

### TTS Engine Configuration

The TTS engine is set by the `ENRS_TTS_ENGINE` environment variable in the backend `.env`. The value is passed directly to FreeSWITCH's `speak` application in the format `engine|voice|text`.

Examples:
- `flite` → uses Flite with the default `kal` voice (offline, no network dependency)
- `PolyglotTTS` → custom TTS module

The voice field in the node config supplements the engine string when the engine supports named voices. If no voice is specified, the engine's default voice is used.

### Database Updates

None.

### ESL Events

FreeSWITCH generates `CHANNEL_EXECUTE` / `CHANNEL_EXECUTE_COMPLETE` for the `speak` application.

### API Calls

None.

### Common Mistakes

- Entering a language tag not in the supported set. The field accepts free text; an unsupported tag causes the TTS engine to fall back to its default.
- Entering a voice name for a TTS engine that does not support named voices (e.g., `flite` ignores the voice field and always uses `kal`).
- Using `${varname}` for a variable that has not been set earlier in the flow — interpolation silently replaces it with an empty string.

### Debugging Steps

1. Verify `ENRS_TTS_ENGINE` is set correctly in `.env`.
2. Check FreeSWITCH logs for `speak` application errors.
3. Test TTS locally: `fs_cli -x "originate loopback/1000 &speak(flite|kal|Hello World)"`

## `gather` — Gather DTMF

**Category:** Input
**Port Strategy:** `branches` (one output port per branch key)
**Terminal:** No

### Purpose

Plays an optional prompt and waits for the caller to press DTMF digits. Routes to different downstream nodes based on the digit(s) pressed. The collected digits are stored in a named session variable for use by downstream `condition` nodes.

### Configuration Fields

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `variable_name` | varName | No | `gather_result` | Session variable name to store collected digits. Must start with a letter or underscore; alphanumeric and underscores only. |
| `max_digits` | integer (1–11) | No | `1` | Maximum number of digits to collect before routing. |
| `timeout_seconds` | integer (1–60) | No | `5` | Seconds to wait for the first digit before timing out. |
| `terminators` | string (max 4 chars) | No | `#` | Keys that end collection before `max_digits` is reached. |
| `prompt_audio_url` | string (`/media/` URI) | No | — | Audio file to play before collecting. Takes precedence over `prompt_text`. |
| `prompt_audio_file_id` | integer | No | — | Alternative to `prompt_audio_url` — ID from `media_files` table. |
| `prompt_text` | string (max 1000 chars) | No | — | TTS text to speak before collecting. Used only if `prompt_audio_url` is absent. |
| `branches` | map: string → node_id | Yes | — | At least one entry required. Special keys: `_default` (any digit not matched above), `invalid` (timeout or no-input). |

### Variables Supported

`prompt_text` supports `${varname}` interpolation.

### Variables Produced

- `variable_name` (default: `gather_result`): set to the collected digit string, or empty string on timeout.

### Validation Rules

- `branches` must have at least one key (Zod refinement on the branches field).
- Branch values must be valid node IDs referencing existing nodes (graph integrity check in the validator).
- `prompt_audio_url` must start with `/media/` if set.
- `prompt_audio_file_id` must reference an existing, active `media_files` row in the same tenant (DB FK check at publish).

### Generated Lua

```lua
local function exec_gather(s, node)
  local br      = node.branches or {}
  local max_d   = node.max_digits      or 1
  local timeout = (node.timeout_seconds or 10) * 1000
  local terms   = node.terminators     or "#"
  local digits

  local pf = resolve_audio(node.prompt_audio_url)
  if pf then
    digits = s:playAndGetDigits(1, max_d, 3, timeout, terms, pf, "", "[0-9#*]+", "", 0)
  else
    local pt = interp(s, node.prompt_text)
    if pt ~= "" then speak(s, pt) end
    digits = s:getDigits(max_d, terms, timeout)
  end

  s:setVariable(node.variable_name or "gather_result", digits or "")
  return br[digits] or br["_default"] or br["invalid"]
end
```

### Runtime Behavior

1. If `prompt_audio_url` resolves to a file, `session:playAndGetDigits()` plays the file while collecting digits simultaneously (barge-in capable).
2. Otherwise, if `prompt_text` is set, TTS is played first, then `session:getDigits()` collects.
3. Collected digits are stored in the named session variable via `session:setVariable()`.
4. The branch lookup order is: exact digit match → `_default` → `invalid`. If none match, the node effectively produces no next-node reference and the executor treats it as a hangup.

### Branch Keys

| Key | When Triggered |
|---|---|
| `"1"` through `"9"`, `"0"`, `"*"`, `"#"` | Exact digit match |
| `_default` | Digit pressed but not in the named branches |
| `invalid` | Timeout or no digit received |

### Database Updates

None.

### ESL Events

`CHANNEL_EXECUTE` / `CHANNEL_EXECUTE_COMPLETE` for `play_and_get_digits` or `read` FreeSWITCH applications.

### API Calls

None.

### Common Mistakes

- Not adding an `_default` or `invalid` branch — callers pressing unexpected digits or timing out will have no route and the call ends abruptly.
- Setting `max_digits=1` but expecting multi-digit PINs. Use `max_digits=8` with `terminators=#` for PIN-style input.
- Referencing `variable_name` in a downstream `condition` node but using a different spelling.

### Debugging Steps

1. Enable `CHANNEL_EXECUTE` logging in FreeSWITCH to verify `play_and_get_digits` parameters.
2. Check `session:getVariable("gather_result")` in subsequent Lua steps.
3. Verify the branch map in the deployed Lua script matches the expected digit strings.

## `condition` — Condition

**Category:** Input
**Port Strategy:** `true_false` (two output ports: `true_node` and `false_node`)
**Terminal:** No

### Purpose

Evaluates a session variable against an expected value using one of eight operators. Routes to `true_node` if the condition is met, `false_node` otherwise. The ENS-specific operators (`ens_pin_valid`, `ens_callback_valid`) make HTTP calls and set session variables as side effects.

### Configuration Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `variable` | varName | Yes | Name of the session variable to evaluate. |
| `operator` | enum | Yes | One of: `==`, `!=`, `contains`, `starts_with`, `ens_pin_valid`, `ens_callback_valid`, `time_of_day`, `day_of_week`. |
| `expected_value` | string (max 256 chars) | Yes | The value to compare against. Supports `${varname}` interpolation. See operator-specific formats below. |
| `true_node` | node_ref | Yes | Node to route to when the condition is satisfied. |
| `false_node` | node_ref | Yes | Node to route to when the condition fails. |

### Operator Behavior

| Operator | Logic | `expected_value` Format |
|---|---|---|
| `==` | Exact string equality (case-sensitive) | Any string |
| `!=` | Not equal | Any string |
| `contains` | Substring match | Any string |
| `starts_with` | Prefix match | Any string |
| `ens_pin_valid` | POST to `/internal/ens/verify-pin`; see below | ENS trigger number or `${destination_number}` |
| `ens_callback_valid` | GET to `/internal/ens/callbacks/authorize`; see below | ENS reply_clid number |
| `time_of_day` | Current time in range (24h, supports overnight) | `"HHMM-HHMM"` e.g. `"0800-1700"` |
| `day_of_week` | Current day in list | Comma-separated day numbers: `0`=Sun…`6`=Sat |

### `ens_pin_valid` Operator Detail

When this operator is selected, the node:

1. Takes `expected_value` as the ENS trigger number (or falls back to the `destination_number` session variable if `expected_value` is empty).
2. Calls `POST /internal/ens/verify-pin` with `{ trigger_number, pin: session.variable }`.
3. If `authorized: true` is returned, calls `GET /internal/ens/lookup?number=<trigger_number>` to resolve configuration metadata.
4. Sets session variables as side effects (see Variables Produced).
5. Routes to `true_node` if authorized, `false_node` otherwise.

### `ens_callback_valid` Operator Detail

When this operator is selected, the node:

1. Takes `expected_value` as the `reply_clid` number.
2. Calls `GET /internal/ens/callbacks/authorize?reply_clid=<expected_value>&caller=<caller_id_number>`.
3. On authorization, sets session variables and routes to `true_node`.
4. On rejection, routes to `false_node`.

### Variables Supported

`expected_value` supports `${varname}` interpolation (resolved before the comparison).

### Variables Produced

For `ens_pin_valid` on success:
- `ens_configuration_id`: the resolved ENS configuration ID (string).
- `ens_blast_clid`: the blast caller ID for the ENS campaign.

For `ens_callback_valid` on success:
- `ens_notification_uuid`: the notification UUID for callback authorization.
- `ens_recording_file`: path to the recording file for playback.
- `ens_delivery_id`: the delivery record ID.

### Validation Rules

- `variable` must be a valid varName (starts with letter/underscore, alphanumeric and underscores).
- `operator` must be one of the eight enumerated values.
- `expected_value` is required (can be empty string only for `ens_pin_valid`/`ens_callback_valid` when using `${destination_number}` fallback).
- Both `true_node` and `false_node` must reference existing nodes.

### Generated Lua

See `backend/src/nodeTypes/registry.js` — `exec_condition` function. Key logic per operator:

```lua
-- == operator
ok = (val == exp)

-- time_of_day operator (supports overnight ranges)
local now_hhmm = tonumber(os.date("%H%M")) or 0
local s_part, e_part = exp:match("^(%d%d%d%d)-(%d%d%d%d)$")
if start_n <= end_n then
  ok = (now_hhmm >= start_n and now_hhmm < end_n)
else
  ok = (now_hhmm >= start_n or now_hhmm < end_n)  -- overnight
end

-- day_of_week operator
local today = tonumber(os.date("%w")) or -1
for d in exp:gmatch("%d+") do
  if tonumber(d) == today then ok = true; break end
end
```

### Runtime Behavior

1. `val = session:getVariable(node.variable)` reads the session variable.
2. `exp = interp(s, node.expected_value)` resolves any `${var}` references.
3. The selected operator logic runs.
4. Returns `node.true_node` if `ok == true`, `node.false_node` otherwise.

For `ens_pin_valid`, two internal API calls may be made (verify-pin + lookup). These are synchronous (blocking) curl calls via `io.popen`.

### Database Updates

None directly. The `verify-pin` and `lookup` endpoints may update delivery/notification state on the backend.

### ESL Events

None.

### API Calls

| Operator | Method | Path |
|---|---|---|
| `ens_pin_valid` | POST | `/api/v1/internal/ens/verify-pin` |
| `ens_pin_valid` (on success) | GET | `/api/v1/internal/ens/lookup?number=...` |
| `ens_callback_valid` | GET | `/api/v1/internal/ens/callbacks/authorize?reply_clid=...&caller=...` |

### Common Mistakes

- Using `ens_pin_valid` with `expected_value` left empty when `destination_number` is not the correct ENS trigger number — the lookup will fail or match the wrong configuration.
- Using `time_of_day` with `expected_value` in `HH:MM-HH:MM` format (with colons) instead of `HHMM-HHMM` (without separators) — the regex `^(%d%d%d%d)-(%d%d%d%d)$` requires no colons.
- Not chaining conditions for AND logic — connect two condition nodes in series for compound conditions.

### Debugging Steps

1. Add a `say` node on the false branch to confirm which path the call takes.
2. For `ens_pin_valid`, check FreeSWITCH logs for the curl call to `/internal/ens/verify-pin` and the response body.
3. For `time_of_day`, verify `os.date("%H%M")` matches expectations in the server's timezone.

## `goto` — Go To Node

**Category:** Flow
**Port Strategy:** `goto_target` (single port keyed `goto`)
**Terminal:** No

### Purpose

Unconditionally jumps to another node in the same flow. Use this to create loops (e.g., replay a menu after invalid input) or to share a common sub-path from multiple branches.

### Configuration Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `target_node_id` | node_ref | Yes | ID of the node to jump to. Must reference an existing node in the same flow. |

### Variables Supported

None.

### Variables Produced

None.

### Validation Rules

- `target_node_id` must reference an existing node in the flow (graph integrity check).
- The validator's terminal-reachability check ensures that the loop created by `goto` still has at least one path that reaches a terminal node. If the `goto` creates a closed loop with no exit, the validator returns an error: `"Node can never reach an end of call — infinite loop with no exit"`.

### Generated Lua

```lua
local function exec_goto(s, node)
  return node.target_node_id
end
```

> **Implementation note:** `goto` is a reserved word in Lua 5.2+. The Lua generator wraps this type in the dispatch table as `["goto"]` (string key) rather than a bare identifier to avoid a parse error.

### Runtime Behavior

1. Returns `node.target_node_id` immediately.
2. The executor's main dispatch loop looks up the returned ID in the node map.
3. The `MAX_STEPS=100` counter in the Lua executor is incremented on every node execution. If the loop runs 100 iterations without reaching a terminal, the executor logs a warning and hangs up.

### Database Updates

None.

### ESL Events

None.

### API Calls

None.

### Common Mistakes

- Creating a loop with no exit path. The validator catches this at publish time — ensure at least one branch of the loop leads to a terminal node.
- Pointing `target_node_id` at the `goto` node itself (a trivial infinite loop). This also fails the terminal-reachability check.

## `set_variable` — Set Variable

**Category:** Recording
**Port Strategy:** `next` (single successor)
**Terminal:** No

### Purpose

Sets a FreeSWITCH channel variable to a static or interpolated value. Use this to prepare variables that downstream `condition` nodes or `webhook` body templates will consume.

### Configuration Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `variable` | varName | Yes | Name of the channel variable to set. Must start with a letter/underscore. |
| `value` | string (max 1024 chars) | Yes | The value to assign. Supports `${varname}` interpolation. |
| `next` | node_ref | Yes | ID of the next node. |

### Variables Supported

`value` supports `${varname}` interpolation — the resolved value is what gets stored.

### Variables Produced

The named `variable` is set to the resolved `value` for the lifetime of the call session.

### Validation Rules

- `variable` must match the varName pattern.
- `value` is required (empty string is accepted).
- `next` is required.

### Generated Lua

```lua
local function exec_set_variable(s, node)
  s:setVariable(node.variable or "unknown_var", interp(s, node.value) or "")
  return node.next
end
```

### Runtime Behavior

1. `interp(s, node.value)` resolves any `${varname}` references.
2. `session:setVariable(name, value)` sets the channel variable.
3. The variable persists for the entire call session.
4. Returns `node.next`.

### Database Updates

None.

### ESL Events

None.

### API Calls

None.

## `hangup` — Hangup

**Category:** Flow
**Port Strategy:** `none` (no output ports)
**Terminal:** Yes

### Purpose

Ends the call, optionally playing a goodbye message first. This is the standard clean termination node for flows that have collected all necessary information.

### Configuration Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `play_audio_url` | string (`/media/` URI) | No | Audio file to play before hanging up. |
| `play_audio_file_id` | integer | No | Alternative to `play_audio_url` — ID from `media_files` table. |

### Variables Supported

None.

### Variables Produced

None.

### Validation Rules

- No required fields. A bare `hangup` node (no goodbye audio) is valid.
- `play_audio_url` must start with `/media/` if set.
- `play_audio_file_id` must reference an existing, active `media_files` row in the same tenant (DB FK check at publish).

### Generated Lua

```lua
local function exec_hangup(s, node)
  local f = resolve_audio(node.play_audio_url)
  if f then s:streamFile(f) end
  s:hangup()
  return nil
end
```

### Runtime Behavior

1. If `play_audio_url` resolves, plays the audio file.
2. Calls `session:hangup()` to terminate the call.
3. Returns `nil` — the executor exits the dispatch loop.

### Database Updates

None.

### ESL Events

FreeSWITCH generates `CHANNEL_HANGUP` and `CHANNEL_HANGUP_COMPLETE` events.

### API Calls

None.

## `transfer` — Transfer

**Category:** Flow
**Port Strategy:** `none` (no output ports)
**Terminal:** Yes

### Purpose

Transfers the call to another extension or dialplan context. Once transferred, the IVR executor loses control of the call — the target dialplan takes over completely.

### Configuration Fields

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `destination` | string (1–128 chars) | Yes | — | Extension number or `${var}`-interpolated destination. |
| `dialplan` | string | No | `XML` | FreeSWITCH dialplan type. Options: `XML`, `inline`, `enum`. |
| `context` | string (max 64 chars) | No | `default` | FreeSWITCH dialplan context name. |

### Variables Supported

`destination` supports `${varname}` interpolation, enabling dynamic routing based on session variables (e.g., `${dept_extension}`).

### Variables Produced

None.

### Validation Rules

- `destination` must be non-empty.
- `dialplan` defaults to `XML`; `context` defaults to `default`.

### Generated Lua

```lua
local function exec_transfer(s, node)
  local dest = interp(s, node.destination) or ""
  s:execute("transfer",
    dest .. " " .. (node.dialplan or "XML") .. " " .. (node.context or "default"))
  return nil
end
```

### Runtime Behavior

1. `interp(s, node.destination)` resolves any variable references.
2. `session:execute("transfer", ...)` hands off call control to the target dialplan.
3. The Lua script exits; no further IVR nodes execute.

### Database Updates

None.

### ESL Events

FreeSWITCH generates `CHANNEL_EXECUTE` for the `transfer` application, followed by standard channel events for the target dialplan.

### API Calls

None.

### Common Mistakes

- Transferring to an extension that does not exist in the target context — the call will receive a 404/USER_NOT_REGISTERED cause.
- Using `inline` dialplan with complex multi-action targets without understanding the inline syntax.

## `record_message` — Record

**Category:** Recording
**Port Strategy:** `next` (single successor)
**Terminal:** No

### Purpose

Plays an optional prompt, plays a recording beep, and records the caller's audio to a file on the FreeSWITCH host. The recording path is stored in a named session variable for downstream use by `ens` or `ens_blast_record` nodes.

### Configuration Fields

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `variable_name` | varName | Yes | — | Session variable to store the recorded file path. |
| `prompt_text` | string (max 1000 chars) | No | — | TTS text to speak before recording. |
| `prompt_audio_url` | string (`/media/` URI) | No | — | Audio file to play before recording. Takes precedence over `prompt_text`. |
| `max_seconds` | integer (1–300) | No | `60` | Maximum recording duration in seconds. |
| `silence_threshold` | integer (10–2000) | No | `500` | Audio level (ms) below which is considered silence. |
| `silence_hits` | integer (1–10) | No | `3` | Consecutive silence chunks before auto-stop. |
| `record_dir` | string (max 512 chars) | No | `${FS_RECORDING_DIR}/ivr` | Directory to write the recording. Defaults to `recordings_dir` global var + `/ivr`. |
| `next` | node_ref | Yes | — | Node to proceed to after recording. |

### Variables Supported

`prompt_text` supports `${varname}` interpolation.

### Variables Produced

- `variable_name`: set to the full absolute path of the recorded WAV file (e.g., `/var/lib/freeswitch/recordings/ivr/ivr_<uuid>_<timestamp>.wav`).

### Validation Rules

- `variable_name` is required.
- `next` is required.
- `prompt_audio_url` must start with `/media/` if set.

### Generated Lua

```lua
local function exec_record_message(s, node)
  local rec_dir = node.record_dir
  if not rec_dir or rec_dir == "" then
    rec_dir = _api:execute("global_getvar", "recordings_dir") or "/var/lib/freeswitch/recordings"
    rec_dir = rec_dir .. "/ivr"
  end
  local fpath = rec_dir .. "/ivr_" .. s:getVariable("uuid") .. "_" .. os.time() .. ".wav"

  local pf = resolve_audio(node.prompt_audio_url)
  if pf then
    s:streamFile(pf)
  else
    local pt = interp(s, node.prompt_text) or ""
    if pt ~= "" then speak(s, pt) end
  end

  s:execute("playback", "tone_stream://%(500,0,640)")  -- recording beep
  s:recordFile(fpath,
    node.max_seconds       or 60,
    node.silence_threshold or 500,
    node.silence_hits      or 3)
  s:setVariable(node.variable_name or "recorded_file_path", fpath)
  return node.next
end
```

### Runtime Behavior

1. Determines the recording directory (`node.record_dir` or global `recordings_dir` + `/ivr`).
2. Constructs the file path using the channel UUID and current timestamp.
3. Plays the optional prompt (audio file takes precedence over TTS).
4. Plays a 500ms tone at 640 Hz (recording beep).
5. Calls `session:recordFile()` — blocks until silence detection triggers, `max_seconds` is reached, or the caller presses `#`.
6. Sets the named session variable to the recorded file path.
7. Returns `node.next`.

### Database Updates

None. The recording file exists on the FreeSWITCH filesystem; it is not tracked in the database unless an `ens` node subsequently triggers a notification referencing it.

### ESL Events

`CHANNEL_EXECUTE` / `CHANNEL_EXECUTE_COMPLETE` for the `record` FreeSWITCH application.

### API Calls

None.

### Common Mistakes

- Using a `record_dir` that the FreeSWITCH process does not have write access to — the `recordFile` call will silently fail and `variable_name` will hold a path to a non-existent file.
- Not connecting the output of `record_message` to an `ens` node — the recording is made but never sent.

### Debugging Steps

1. Verify `recordings_dir` is set in FreeSWITCH global variables: `fs_cli -x "global_getvar recordings_dir"`
2. Check that the directory exists and is writable by the FreeSWITCH process.
3. After a test call, check the filesystem for the expected file path.

## `webhook` — Webhook

**Category:** Integrations
**Port Strategy:** `next` (single successor)
**Terminal:** No

### Purpose

Posts a JSON payload to an external HTTP endpoint as a fire-and-forget side effect during the call flow. The response is not parsed or acted upon. Use this to notify external systems (ticketing systems, monitoring platforms) of call events.

### Configuration Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `url` | string (1–2048 chars) | Yes | The HTTP endpoint to POST to. Must be a full URL (no SSRF restrictions on scheme — this node contacts external URLs, unlike `audio_url` which is local-only). Supports `${varname}` interpolation. |
| `body_template` | string (max 4000 chars) | No | JSON body template. Supports `${varname}` interpolation. If omitted, sends `{}`. |
| `next` | node_ref | Yes | Node to proceed to after the webhook fires. |

### Variables Supported

Both `url` and `body_template` support `${varname}` interpolation.

### Variables Produced

None. The response body is discarded.

### Validation Rules

- `url` is required and must be non-empty.
- `next` is required.
- There is no URL scheme restriction for webhooks (unlike `audio_url`). Exercise caution with untrusted URLs.

### Generated Lua

```lua
local function exec_webhook(s, node)
  local url = interp(s, node.url) or ""
  if url == "" then
    freeswitch.consoleLog("ERR", "[ivr_executor] webhook node: empty url — skipping\n")
    return node.next
  end
  local body = interp(s, node.body_template) or "{}"
  local safe_body = body:gsub("'", "\\'")
  local cmd = string.format(
    "curl -s -m %d -X POST -H 'Content-Type: application/json' -d '%s' '%s' 2>/dev/null",
    HTTP_TIMEOUT, safe_body, url)
  local h = io.popen(cmd)
  if h then h:close() end
  return node.next
end
```

The `HTTP_TIMEOUT` constant is set in the Lua generator (default: 5 seconds). The call does **not** wait for the response beyond the timeout.

### Runtime Behavior

1. Interpolates `url` and `body_template` with session variables.
2. Executes a `curl` POST via `io.popen` — this runs in a subprocess and does not block the FreeSWITCH thread beyond the timeout.
3. Discards the response.
4. Returns `node.next` immediately.

> **Note:** Webhook does not use the internal API authentication header — it contacts arbitrary external URLs. Do not use it to call the app's own internal API; use ENS/ERS nodes for that.

### Database Updates

None.

### ESL Events

None.

### API Calls

External POST to the configured `url`. Not routed through the internal API.

### Common Mistakes

- Using webhook to call the app's own `/api/v1/internal/*` endpoints — use `ens` or `ers` nodes instead.
- Including the caller's PIN or sensitive data in the body template — the body is logged in FreeSWITCH logs and the remote server.
- Setting a very long timeout that delays the call flow.

## `ens` — Trigger ENS

**Category:** Emergency
**Port Strategy:** `next_optional` (single `next` port, shown only if `next` is set)
**Terminal:** No (but can act as a flow-ender if `next` is absent)

### Purpose

Triggers an ENS (Emergency Notification System) broadcast campaign. The caller is identified, the ENS configuration is resolved, and a blast notification is started asynchronously. The caller can optionally be routed to a continuation node after the blast is confirmed started.

### Configuration Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `ens_configuration_id` | integer | One of `ens_configuration_id` or `ens_config_var` required | FK to `ens_configurations`. The ENS configuration to trigger. |
| `ens_config_var` | varName | One of `ens_configuration_id` or `ens_config_var` required | Session variable name holding the ENS configuration ID at runtime (set by `condition` node with `ens_pin_valid` operator). |
| `recording_file_var` | varName | No | Session variable holding the path to a recording file (set by a prior `record_message` node) to include in the blast. |
| `next` | node_ref | No | Continuation node after the blast is triggered. If absent, the port is not shown and the flow ends here after triggering. |

### Variables Supported

None directly (configuration is resolved from session variables, not interpolated strings).

### Variables Produced

- `ens_notification_uuid`: set to the notification UUID returned by the backend on successful blast trigger.

### Validation Rules

- Either `ens_configuration_id` or `ens_config_var` must be set (enforced by `superRefine` in `AnyNodeSchema`).
- `ens_configuration_id` must reference an existing, active `ens_configurations` row in the same tenant (DB FK check at publish).

### Generated Lua

```lua
local function exec_ens(s, node)
  local cfg_id = node.ens_configuration_id
  if not cfg_id or cfg_id == 0 then
    cfg_id = tonumber(s:getVariable(node.ens_config_var or "ens_configuration_id") or "")
  end
  if not cfg_id then
    freeswitch.consoleLog("ERR", "[ivr_executor] ens node: could not resolve configuration_id — skipping\n")
    return node.next
  end

  local caller_number = s:getVariable("caller_id_number") or ""
  local recording_file
  if node.recording_file_var and node.recording_file_var ~= "" then
    local rf = s:getVariable(node.recording_file_var) or ""
    if rf ~= "" then recording_file = rf end
  end

  local d = post("/ens/notifications", {
    configuration_id = cfg_id,
    triggered_via    = "PHONE",
    caller_number    = caller_number ~= "" and caller_number or nil,
    recording_file   = recording_file,
  })

  if d and d.notification_uuid then
    s:setVariable("ens_notification_uuid", d.notification_uuid)
  end

  return node.next
end
```

### Runtime Behavior

1. Resolves `cfg_id` from `ens_configuration_id` field or from the `ens_config_var` session variable.
2. If neither resolves, logs an error and returns `node.next` (does not crash the call).
3. Reads `recording_file_var` session variable if configured.
4. POSTs to `/internal/ens/notifications` to start the blast.
5. Sets `ens_notification_uuid` session variable on success.
6. Returns `node.next` (or `nil` if `next` is absent, ending the call).

### Database Updates

The backend creates an `ens_notifications` row and starts the campaign engine.

### ESL Events

None generated by this node. The ENS campaign engine generates ESL `originate` commands to call contacts.

### API Calls

| Method | Path |
|---|---|
| POST | `/api/v1/internal/ens/notifications` |

- `ers_incident_uuid`: set to the incident UUID.

### Validation Rules

- `ers_configuration_id` is required (Zod `optionalConfigId` coerces empty string/null/0 to `undefined` for draft nodes, but the graph-integrity validator warns if the value is absent).

**Category:** Emergency
**Port Strategy:** `branches` (three ports: `primary`, `secondary`, `full`)
**Terminal:** No

### Purpose

Checks the live occupancy of ERS tier conference rooms and routes the caller to the appropriate tier or overflow path. Occupancy is determined by the live FreeSWITCH conference member count, not the incident status in the database.

### Configuration Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `ers_configuration_id` | integer | Yes | FK to `ers_configurations`. |

| Key | Condition |

### Variables Supported

None.

### Variables Produced

None.

### Validation Rules

- The graph-integrity validator warns (not errors) if any of the three branch keys (`primary`, `secondary`, `full`) are not connected.

None (read-only status check).

## `ers_overflow_wait` — ERS Overflow Wait

**Category:** Emergency
**Port Strategy:** `next` (single successor — fallback path)
**Terminal:** No

### Purpose

Places the caller in a queue, plays hold music or an announcement, and polls tier occupancy. When a tier becomes free, the caller is automatically bridged into the conference. If the maximum wait time is exceeded or the queue entry is cancelled, routes to the fallback `next` node.

### Configuration Fields

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `ers_configuration_id` | integer | Yes | — | FK to `ers_configurations`. |
| `hold_prompt_text` | string (max 1000 chars) | No | — | TTS hold announcement played while in queue. |
| `hold_audio_url` | string (`/media/` URI) | No | — | Audio file for hold music/announcement. Takes precedence over `hold_prompt_text`. |
| `max_wait_seconds` | integer (10–3600) | No | `300` | Maximum queue wait time before routing to fallback. |
| `next` | node_ref | Yes | — | Fallback node when wait cap is hit or queue is cancelled. |

### Variables Supported

`hold_prompt_text` supports `${varname}` interpolation.

### Variables Produced

None.

### Validation Rules

- `ers_configuration_id` is warned if absent on draft nodes.
- `next` is required.

### Generated Lua

```lua
local function exec_ers_overflow_wait(s, node)
  local enq = post("/ers/overflow/enqueue", { ... })
  -- ... play hold prompt ...
  local deadline = os.time() + max_wait
  while s:ready() and os.time() < deadline do
    local d = get("/ers/overflow/poll?queue_id=" .. tostring(enq.queue_id))
    if d and d.ready and d.conference_room then
      s:execute("conference", d.conference_room .. "@default")
      return nil
    end
    if d and d.cancelled then return node.next end
    s:execute("playback", "silence_stream://3000")  -- 3s wait between polls
  end
  return node.next
end
```

### Runtime Behavior

1. Enqueues the caller via POST to `/internal/ers/overflow/enqueue`.
2. Plays hold prompt (audio file or TTS).
3. Enters a polling loop: every 3 seconds, GETs `/internal/ers/overflow/poll?queue_id=<id>`.
4. On `ready: true` and `conference_room` in the response: bridges caller into the conference (`session:execute("conference", ...)`), marks incident complete, returns `nil`.
5. On `cancelled: true`: returns `node.next` (fallback).
6. On `max_wait_seconds` exceeded: returns `node.next` (fallback).
7. On caller disconnect (`session:ready()` returns false): exits the loop.

### Database Updates

Creates a queue entry via the backend enqueue endpoint.

### API Calls

| Method | Path |
|---|---|
| POST | `/api/v1/internal/ers/overflow/enqueue` |
| GET | `/api/v1/internal/ers/overflow/poll?queue_id=<id>` |
| POST | `/api/v1/internal/ers/incidents/:uuid/complete` |

## `ens_blast_record` — ENS Blast (PIN + Record)

**Category:** Emergency
**Port Strategy:** `next` (single successor)
**Terminal:** No

### Purpose

A self-contained ENS trigger node that handles the full blast workflow in sequence: PIN gate (3 attempts), record a voice message, and broadcast to all contacts. This is the preferred node for phone-triggered ENS notification when you want a single-step flow without separate `gather` + `condition` + `record_message` + `ens` nodes.

### Configuration Fields

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `ens_configuration_id` | integer | No | — | FK to `ens_configurations`. If absent, resolved from the dialed number via `/internal/ens/lookup`. |
| `pin_prompt_text` | string (max 1000 chars) | No | "Please enter your authorization PIN followed by pound." | TTS prompt for PIN entry. |
| `record_prompt_text` | string (max 1000 chars) | No | "Record your emergency message after the tone. Press pound when finished." | TTS prompt before recording. |
| `max_record_seconds` | integer (5–300) | No | `120` | Maximum recording duration. |
| `next` | node_ref | Yes | — | Node to route to after blast is confirmed started. |

### Variables Supported

`pin_prompt_text` and `record_prompt_text` support `${varname}` interpolation.

### Variables Produced

- `ens_notification_uuid`: set to the notification UUID returned by the backend on successful blast trigger.

### Validation Rules

- `next` is required (validator warns if not connected).
- `ens_configuration_id` is optional — resolved from the dialed number at runtime if absent.

### Generated Lua

```lua
local function exec_ens_blast_record(s, node)
  local dest = s:getVariable("destination_number") or ""

  -- PIN gate with 3 attempts
  local authorized = false
  for attempt = 1, 3 do
    local prompt = interp(s, node.pin_prompt_text)
    if prompt == "" then prompt = "Please enter your authorization PIN followed by pound." end
    speak(s, prompt)
    local pin = s:getDigits(8, "#", 10000)
    local verify = post("/ens/verify-pin", { trigger_number = dest, pin = pin or "" })
    if verify and verify.authorized then
      authorized = true; break
    end
    speak(s, "Invalid PIN.")
  end
  if not authorized then
    speak(s, "Maximum authorization attempts exceeded. Goodbye.")
    s:hangup("CALL_REJECTED")
    return nil
  end

  -- Resolve configuration_id
  local cfg_id = node.ens_configuration_id
  if not cfg_id then
    local lookup = get("/ens/lookup?number=" .. url_encode(dest))
    if lookup and lookup.success and lookup.data then
      cfg_id = lookup.data.configuration_id
    end
  end

  -- Record message
  local fpath = rec_dir .. "/ens/ens_" .. tostring(cfg_id) .. "_" .. os.time() .. ".wav"
  -- ... speak prompt, play beep, recordFile ...

  -- Blast
  local d = post("/ens/notifications", {
    configuration_id = cfg_id,
    triggered_via    = "PHONE",
    caller_number    = s:getVariable("caller_id_number") or nil,
    recording_file   = fpath,
  })

  if d and d.notification_uuid then
    s:setVariable("ens_notification_uuid", d.notification_uuid)
    speak(s, "Your emergency notification is now being sent to all contacts.")
  end
  return node.next
end
```

### Runtime Behavior

1. Reads `destination_number` to identify the ENS trigger number.
2. Presents the PIN prompt up to 3 times. Calls `POST /internal/ens/verify-pin` after each attempt.
3. On 3 failures, speaks "Maximum authorization attempts exceeded" and hangs up (CALL_REJECTED cause).
4. If `ens_configuration_id` is not configured in the node, calls `GET /internal/ens/lookup` to resolve from the dialed number.
5. Speaks the record prompt and plays a recording beep.
6. Calls `session:recordFile()` — blocks until silence or max time.
7. POSTs to `/internal/ens/notifications` with the recording file path.
8. Sets `ens_notification_uuid` and speaks confirmation.
9. Returns `node.next`.

### Database Updates

Creates an `ens_notifications` row and starts the campaign engine via the backend.

### API Calls

| Method | Path |
|---|---|
| POST | `/api/v1/internal/ens/verify-pin` (up to 3 times) |
| GET | `/api/v1/internal/ens/lookup?number=...` (if `ens_configuration_id` not set) |
| POST | `/api/v1/internal/ens/notifications` |

### Common Mistakes

- Placing this node in a flow that already has a `gather` + `condition ens_pin_valid` — the PIN will be collected twice. Use either this node or the separate nodes, not both.

### Validation Rules

- `ers_configuration_id` is warned if absent on draft nodes.
- Both `true_node` and `false_node` must reference existing nodes.

### Generated Lua

```lua
local function exec_ens_playback_gate(s, node)
  local caller = s:getVariable("caller_id_number") or ""
  local d = get("/ers/playback/authorize?configuration_id=" ..
                tostring(node.ers_configuration_id or 0) ..
                "&caller=" .. url_encode(caller))

  if not d or not d.authorized then
    freeswitch.consoleLog("WARN", "[ivr_executor] ens_playback_gate: rejected caller " .. caller .. "\n")
    return node.false_node
  end

  if d.recording_file then
    s:streamFile(d.recording_file)
  else
    local msg = interp(s, node.no_message_text)
    if msg == "" then msg = "There is no active emergency message at this time." end
    speak(s, msg)
  end
  return node.true_node
end
```

### Runtime Behavior

1. Reads `caller_id_number` from the session.
2. GETs `/internal/ers/playback/authorize?configuration_id=<id>&caller=<caller_number>`.
3. If `authorized: false` or API error: logs WARN and routes to `false_node`.
4. If `authorized: true` and `recording_file` is present: streams the recording file via `session:streamFile()`.
5. If `authorized: true` but no recording file: speaks `no_message_text` (or the default message).
6. Routes to `true_node` on authorization.

### Database Updates

None directly. The backend authorization check may log the callback access event.

### ESL Events

`CHANNEL_EXECUTE` / `CHANNEL_EXECUTE_COMPLETE` for the `playback` application if a recording is streamed.

### API Calls

| Method | Path |
|---|---|
| GET | `/api/v1/internal/ers/playback/authorize?configuration_id=<id>&caller=<caller>` |

### Common Mistakes

- Confusing `ers_configuration_id` with `ens_configuration_id` — this node requires the ERS configuration ID, not the ENS one. The field label in the UI reads "ERS Configuration."
- Not connecting `false_node` — unauthorized callers will have no route.

### Debugging Steps

1. Check FreeSWITCH logs for the `ens_playback_gate` WARN line to confirm the caller was rejected and why.
2. Verify the caller's number is on the ENS delivery list for the associated notification.
3. Confirm the recording file path returned by the authorize endpoint is accessible on the FreeSWITCH filesystem.
