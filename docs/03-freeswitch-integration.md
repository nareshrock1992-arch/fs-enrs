# FreeSWITCH Integration

## ESL Connection

The backend maintains a single persistent TCP connection to FreeSWITCH's Event Socket Library (ESL) using the `modesl` npm package. This connection carries two types of traffic:

- **Inbound events**: FreeSWITCH pushes call lifecycle and conference events to the backend.
- **Outbound commands**: The backend sends `bgapi` commands to FreeSWITCH to control calls and conferences.

### Connection Parameters

| Environment Variable | Default | Notes |
|---|---|---|
| `ESL_HOST` | `127.0.0.1` | FreeSWITCH host |
| `ESL_PORT` | `8021` | ESL TCP port (mod_event_socket) |
| `ESL_PASSWORD` | `ClueCon` | ESL password from `event_socket.conf.xml` |
| `ESL_RECONNECT_MS` | `3000` | Base reconnect interval; doubles per attempt up to 30 s |

### Event Subscriptions

Immediately after connecting, the backend subscribes to exactly these event types:

```
CUSTOM conference::maintenance
CHANNEL_HANGUP
CHANNEL_ANSWER
CHANNEL_CREATE
CHANNEL_BRIDGE
DTMF
RECORD_STOP
```

No other events are subscribed. The `CUSTOM conference::maintenance` subscription covers all conference room lifecycle and per-member state changes.

## ESL Command Execution

All commands are sent via `bgapi`. The `eslCommand(cmd)` export wraps `eslCommandTimeout(cmd, 10_000)` — a 10-second hard timeout. If the ESL connection drops between the connected-check and the send, the timeout fires and the caller receives a rejected Promise. This prevents HTTP request handlers from hanging indefinitely behind a stale ESL connection.

Commands that are issued frequently or need tighter guarantees pass an explicit `timeoutMs` to `eslCommandTimeout` directly (e.g., `conference list` uses an 8-second timeout, `conference <n> xml_list` uses a 5-second timeout).

## Conference Room Naming

ERS conference rooms follow a strict naming pattern generated by the Lua script:

```
ers_<config_id>_<unix_epoch>
```

Example: `ers_42_1720000000`

The room name is generated by Lua before the API call, passed to `POST /internal/ers/incidents` as `conference_room`, and stored in `ers_incidents.conference_room`. The backend ring-all logic uses this room name to route subsequent responders into the same conference.

For **slot-based routing** (used when `max_concurrent_conferences` > 1):

- Primary tier slot N: `ers_<config_id>_pN` (e.g., `ers_42_p1`, `ers_42_p2`)
- Secondary tier slot N: `ers_<config_id>_sN` (e.g., `ers_42_s1`)
- Callers beyond all configured slots are placed in `ers_queues` with status `QUEUED`.

## Ring-All Originate

When the ERS ring-all service is triggered, the backend originates simultaneous outbound call legs to all responders in a tier using FreeSWITCH `bgapi originate`. The command pattern is:

```
bgapi originate {
  origination_caller_id_number=<initiator_number>,
  origination_caller_id_name=<initiator_number>,
  ignore_early_media=true,
  originate_timeout=30,
  origination_uuid=<pre_assigned_uuid>
}user/2001 &conference(ers_42_p1@default)
```

### CallerID Passthrough

`origination_caller_id_number` is set to the **initiator's** number. This means the responder's phone displays who called the ERS, not an internal system number. This is intentional and is the source of a critical participant-tracking subtlety (see `trackParticipant` in the ESL Event Processing document).

### Destination Number for Tracking

Because `Caller-Caller-ID-Number` in the conference join event carries the initiator's number (not the responder's), the responder's actual extension is identified via `Caller-Destination-Number`. The `trackParticipant` function uses destNum-first resolution to correctly attribute join/leave events to the responder, not the initiator.

## ENS Campaign Originate

`originateCampaignCall()` in `eslService.js` issues outbound calls for ENS blast deliveries. The dial string is resolved by `dialResolver.js` based on `ENS_ORIGINATE_MODE`:

| Mode | Dial String |
|---|---|
| `user` (lab/extension default) | `user/<extension>` |
| `gateway` (production SIP) | `sofia/gateway/<gatewayName>/<number>` |

Key channel variables set on every campaign call:

```
origination_uuid=<pre_assigned_call_uuid>
origination_caller_id_number=<clid>
origination_caller_id_name=Emergency
ignore_early_media=true
originate_timeout=30
enrs_campaign_id=<campaignId>
enrs_dest_id=<deliveryId>
```

The `origination_uuid` is assigned before the originate command so that `CHANNEL_ANSWER` and `CHANNEL_HANGUP` events can be correlated to the specific campaign delivery without a lookup.

If `playbackFile` is provided, the call immediately plays that file (`&playback(<path>)`). If null, the call is parked (`&park()`) for media injection.

## Conference ESL Commands

All conference control operations use the pattern `conference <name> <action> [args]`. The following exports are available from `eslService.js`:

| Function | ESL Command | Purpose |
|---|---|---|
| `confPlay(name, path)` | `conference <n> play <path>` | Play audio to all conference members |
| `confKick(name, id)` | `conference <n> kick <id>` | Remove a member |
| `confMute(name, id)` | `conference <n> mute <id>` | Mute a member |
| `confUnmute(name, id)` | `conference <n> unmute <id>` | Unmute a member |
| `confDeaf(name, id)` | `conference <n> deaf <id>` | Deafen a member |
| `confUndeaf(name, id)` | `conference <n> undeaf <id>` | Undeafen a member |
| `confLock(name)` | `conference <n> lock` | Lock the conference |
| `confUnlock(name)` | `conference <n> unlock` | Unlock the conference |
| `confVolumeIn(name, id, level)` | `conference <n> volume_in <id> <level>` | Adjust member's input volume |
| `confVolumeOut(name, id, level)` | `conference <n> volume_out <id> <level>` | Adjust member's output volume |
| `confEnergy(name, id, level)` | `conference <n> energy <id> <level>` | Set member energy threshold |
| `confFloor(name, id)` | `conference <n> floor <id>` | Grant floor to member |
| `confTransfer(name, id, ext)` | `conference <n> transfer <id> <ext> XML default` | Transfer member to extension |
| `confRecord(name, path)` | `conference <n> record <path>` | Start recording to path |
| `confRecordStop(name, path)` | `conference <n> norecord <path>` | Stop recording specific file |
| `confRecordStopAll(name)` | `conference <n> norecord all` | Stop all recordings (fallback) |
| `confTerminate(name)` | `conference <n> kick all` | Empty the room |
| `confSay(name, text)` | `conference <n> say <text>` | TTS broadcast to conference |
| `confInvite(name, dialString)` | `conference <n> bgdial <dialString>` | Invite a dial string into conference |
| `confModerator(name, id)` | `conference <n> moderator <id>` | Toggle moderator role |
| `confList(name)` | `conference <n> list` | List members (text format) |
| `confXmlList(name)` | `conference <n> xml_list` | List members (XML, authoritative) |
| `getConferenceMemberCount(name)` | `conference <n> count` | Live member count |

After every control command (mute, unmute, deaf, etc.), `syncConferenceFromXml(confName)` is called 300 ms later to reconcile the in-memory registry with FreeSWITCH's authoritative state. This corrects any drift between what the registry expects and what FreeSWITCH actually applied.

## XML Dialplan Generation

The deployment pipeline generates a single file `enrs_ivr.xml` containing one `<extension>` block per bound emergency number. Each extension routes the matched destination number to the Lua executor:

```xml
<context name="default">
  <extension name="enrs_ivr_1800EMERGENCY">
    <condition field="destination_number" expression="^1800EMERGENCY$">
      <action application="lua" data="ivr_executor.lua"/>
    </condition>
  </extension>
  <!-- one block per bound number -->
</context>
```

The `<context name="default">` wrapper is included or omitted depending on the detected dialplan layout (see the XML Generation Pipeline document). In nested-include layouts, the file is written as a bare `<extension>` fragment inside FreeSWITCH's already-open default context.

## Lua Script

### `ivr_executor.lua`

The single generic Lua script deployed to `getExecutorLuaFile()` (default: `/usr/share/freeswitch/scripts/ivr_executor.lua`). Generated by `luaGenerator.js` on every deploy. Handles all IVR flow types — ERS, ENS, plain IVR, and all 17 node types — by fetching the flow graph from the backend at call time.

### Legacy Standalone Scripts

Two additional Lua scripts handle non-IVR ERS/ENS use cases and are not managed by the IVR deployment pipeline. They reside in the FreeSWITCH scripts directory and are configured in FreeSWITCH dialplan entries manually:

| Script | Purpose |
|---|---|
| `ers_conference_bridge.lua` | Direct ERS conference trigger (legacy, pre-IVR) |
| `ens_blast_trigger.lua` | PIN-gate + record + blast trigger (legacy) |
| `ens_playback_handler.lua` | Authorized playback of latest ENS recording |

All three use `io.popen(curl ...)` for HTTP and the same `X-Internal-Key` authentication.

## reloadxml and Extension Verification

After writing the dialplan XML file, the deployment engine sends `bgapi reloadxml` via ESL. This causes FreeSWITCH to re-parse all XML configuration files. However, a `+OK` response from `reloadxml` only confirms that parsing succeeded — it does not confirm that the extension was merged into the routing context that live calls use.

The extension verification step addresses this:

```
xml_locate dialplan context name default
```

This command resolves the default dialplan context exactly as a real call would. The response XML is scanned for the string `enrs_ivr_<number>`. If not found, the deployment is marked as failed with a diagnostic message directing the operator to check `dialplan/default.xml`'s include pattern.

Verification is retried 3 times with a 500 ms delay between attempts to account for the race condition between `reloadxml`'s ESL response and FreeSWITCH completing its internal re-parse.

If ESL is offline during deployment, the files are still written successfully, `reloadxml` is skipped (with a warning), and verification is also skipped. The dialplan will load on the next FreeSWITCH restart.

## Recording Paths

All recordings are stored under the base recording directory (controlled by `FS_RECORDINGS_DIR`, `FS_RECORDING_DIR`, or `ENRS_REC_DIR` in order of precedence; default `/var/lib/freeswitch/recordings`).

| Subdirectory | Type | Populated by |
|---|---|---|
| `/ers/` | ERS conference recordings | `confRecord()` via ESL, initiated by operator from UI |
| `/ens/` | ENS blast message recordings | `s:recordFile()` in `ens_blast_record` Lua node |
| `/ivr/` | IVR `record_message` node recordings | `s:recordFile()` in `record_message` Lua node |
| `/manual/` | Operator-initiated conference recordings | `confRecord()` when no ERS incident is active |

Recording type is inferred by the backend from the file path: a path containing `/ers/` is type `ERS`, `/ens/` is type `ENS`, `/ivr/` is type `IVR`, `/manual/` is type `MANUAL`. This inference runs in both the `start-recording` ESL event handler and the `RECORD_STOP` event handler.

## SIP Gateways

SIP gateway definitions are stored in the `sip_gateways` table with columns for `name`, `host`, `port`, `username`, `password`, `realm`, and `tenant_id`. The gateway deployment service (`gatewayDeployment.js`) generates FreeSWITCH gateway XML fragments via `gatewayXmlGenerator.js` and deploys them to `FS_SIP_PROFILE_DIR` (default `/etc/freeswitch/sip_profiles/`).

Gateway selection for outbound calls is resolved by `dialResolver.js` which checks (in order):

1. Contact-specific `gatewayId` override.
2. Tenant-default gateway.
3. Fall through to `user/<extension>` (local SIP registration).

These variables are baked into the generated `ivr_executor.lua` at deploy time by `deployLuaExecutor()`:

| Variable | Purpose | Default |
|---|---|---|
| `ENRS_API_URL` | Base URL for internal API calls | `http://127.0.0.1:4100` |
| `INTERNAL_API_KEY` | `X-Internal-Key` value | (required in production) |
| `FS_TTS_ENGINE` | TTS engine string `engine|voice` | `flite|kal` |

These variables are consumed by the standalone Lua scripts at runtime (read from FreeSWITCH global variables or `io.popen` environment):

| Variable | Purpose |

| `ENRS_TTS_VOICE` | TTS voice name |
| `ENRS_ERS_REC_DIR` | ERS recording directory |
| `ENRS_REC_DIR` | ENS recording directory |
